# 旅行短视频剪辑导演助手 SOP v1

## 1. 文档目标

本文档用于规范一期 Web App 的业务操作流程，确保系统能够稳定实现以下目标：

- 输入素材脑图、素材卡片和项目目标。
- 输出不创造素材、可直接照着剪的分镜脚本。
- 产出可用于剪辑和发布的配套方案，包括节奏建议、字幕建议和发布文案。

适用范围：

- 一期 MVP：旅行短视频剪辑导演助手。
- 输出对象：运营、剪辑师、产品、开发、测试。

## 2. 产品定位

- 产品定位：旅行短视频剪辑导演助手。
- 一期核心能力：生成剪辑脚本，而不是自动生成成片。
- 核心价值：帮助用户把“很多素材但不会组织”的问题，转化成“有结构、有主题、有节奏的可执行剪辑方案”。

## 3. 参与角色

- 用户：录入项目和素材信息，选择叙事主题，确认分镜并导出。
- 系统：分析素材、生成主题、生成分镜、生成节奏与发布方案。
- 剪辑师：根据系统导出的分镜脚本在剪映、Premiere Pro 或 DaVinci Resolve 中完成精剪。

## 4. 一期标准作业流程

## 4.1 项目创建

目标：建立项目基础信息，明确平台和视频目标。

输入项：

- 项目名称
- 目的地
- 发布平台
- 目标时长
- 视频类型
- 风格偏好
- 路线信息

处理规则：

- 平台是核心变量，需影响后续分镜节奏、字幕风格和发布方案。
- 目标时长必须作为分镜总时长约束。

输出：

- 项目基础档案
- 项目唯一 ID

验收标准：

- 项目可被保存、编辑、复制。
- 平台、时长、风格字段完整且后续流程可复用。

## 4.2 素材录入与整理

目标：把非结构化素材信息整理成结构化素材表。

允许的录入方式：

- 粘贴素材脑图
- 手动录入素材卡片
- 上传截图辅助描述
- 批量导入文件名列表

一期原则：

- 允许半人工整理，不强求全自动视频识别。
- 所有用于分镜的素材都必须生成 `asset_id`。

标准素材字段：

| 字段 | 说明 |
| --- | --- |
| `asset_id` | 素材唯一 ID |
| `location` | 地点归属 |
| `scene` | 画面内容 |
| `media_type` | 视频、照片、航拍、手机、相机等 |
| `shot_type` | 大景、中景、特写、人物、动物、转场等 |
| `emotion_tags` | 冷、静、荒野、童话、烟火等 |
| `visual_tags` | 冷蓝、暖灯、白雪、橙色等 |
| `information_density` | 高、中、低 |
| `suggested_duration` | 建议镜头时长 |
| `function` | 开头钩子、节奏刺点、主高潮、收尾等 |

`shot_type` 补充说明：

- `中景` 用于表示同时包含景别信息和人物或主体关系的镜头，适合承接环境与主体之间的叙事连接。

处理规则：

- `asset_id` 不允许为空。
- `location` 必填，用于保证地点连续性。
- `information_density` 必须参与镜头时长分配。
- 无法确认内容的素材不可直接进入最终分镜。

输出：

- 结构化素材表
- 按地点或场景聚合后的素材分组

验收标准：

- 每条素材均可被追溯。
- 分镜生成前能够按地点、场景、镜头类型筛选素材。

## 4.3 素材脑图整理

目标：建立清晰的地点、路线和场景结构，避免叙事混乱。

推荐分组维度：

- 地点
- 路线顺序
- 场景
- 镜头功能

处理规则：

- 默认优先按路线顺序组织。
- 同一地点的素材可继续按场景拆分。
- 脑图节点需允许用户拖拽调整顺序。

输出：

- 树状素材脑图
- 可编辑素材组

验收标准：

- 用户能清楚识别素材的地点归属和叙事顺序。
- 脑图结构可直接供主题生成和分镜生成使用。

## 4.4 叙事主题生成与确认

目标：先定主题，再生成分镜，避免素材堆叠。

输入：

- 项目基础档案
- 素材表
- 素材脑图
- 平台信息

系统输出：

- 3 至 5 个叙事主题方向

主题示例维度：

- 情绪片
- 攻略片
- 城市氛围片
- 路线纪实片
- 人物陪伴感 Vlog

处理规则：

- 主题必须基于现有素材能力生成，不能脱离素材本身。
- 每个主题需给出一句话定位、核心情绪、预期节奏和适配平台理由。

输出：

- 用户选定的主主题

验收标准：

- 用户可在多个方向中快速判断最合适的叙事方式。
- 选定主题后进入分镜生成流程。

## 4.5 分镜生成

目标：输出逐秒可执行的时间线脚本。

输入：

- 选定主题
- 素材表
- 目标时长
- 平台信息

标准输出字段：

| 字段 | 说明 |
| --- | --- |
| `start_time` | 起始时间 |
| `end_time` | 结束时间 |
| `asset_id` | 引用素材 ID |
| `shot_description` | 镜头描述 |
| `function` | 功能位 |
| `rhythm` | 节奏说明 |
| `subtitle` | 字幕建议 |

核心规则：

- 不得创造素材表中不存在的镜头。
- 每个分镜必须绑定 `asset_id`。
- 必须遵守真实路线和地点归属连续性。
- 按信息量和情绪强度分配镜头时长。
- 按开头、1/4、1/2、3/4、结尾设计爆点。
- 如已选定 BGM 或节拍模板，镜头起止时间应优先贴合节拍点，而不是只按均分秒数切割。

分镜校验项：

- 是否存在未绑定 `asset_id` 的镜头
- 是否出现跨地点错误跳转
- 总时长是否在目标区间内
- 是否存在明显冗余或节奏断裂
- 是否出现明显偏离节拍点的卡点错误

输出：

- 可直接执行的分镜时间线
- Markdown、JSON、YAML、PDF 导出内容

验收标准：

- 剪辑师可直接照脚本完成粗剪或精剪。
- 人工阅读时无需再猜素材对应关系。

## 4.6 音乐与节奏方案

目标：给出可执行的节奏设计，而不是仅推荐歌名。

输入：

- 平台信息
- 分镜时间线
- 素材动势
- 风格偏好
- 可选 BGM 或 BGM 风格

输出：

- BGM 风格建议
- 快慢节奏说明
- 暗场位置建议
- 慢推、快闪、转场建议
- 节拍点建议

处理规则：

- 音乐建议需服务于结构节奏，而非独立输出。
- 对照片素材应给出慢推、缩放、停留建议。
- 当用户已选定具体 BGM 时，系统应像剪映一样优先识别可用节拍点，例如 `节拍 1`、`节拍 2` 或强拍/弱拍序列。
- 生成剪辑秒数时，优先以节拍点为时间锚点，再结合素材信息量和情绪强度微调镜头长度。
- 若尚未选定具体 BGM，一期可先输出“节拍模板”，例如每 0.5 秒、1 秒或 2 秒的卡点结构，供后续替换歌曲时重算。

验收标准：

- 剪辑师可根据节奏方案落地时间线。
- 不同平台可得到差异化节奏建议。
- 节奏方案与分镜秒数之间不存在明显矛盾。

## 4.7 发布方案生成

目标：在分镜完成后，直接生成发布所需文案。

输入：

- 平台信息
- 视频主题
- 分镜摘要
- 地点和亮点信息

输出：

- 标题
- 短标题
- 描述
- 标签
- 封面建议

处理规则：

- 同一视频在不同平台要输出不同风格文案。
- 标题要结合地点符号、情绪钩子和平台表达习惯。

验收标准：

- 用户剪完后可直接进入发布环节。
- 文案与视频主题保持一致，不空泛、不跑题。

## 4.8 导出与交付

目标：将结果沉淀为可协作、可复用的标准资料。

导出格式：

- Markdown
- JSON
- YAML
- PDF

使用场景：

- Markdown：人工阅读与协作修改
- JSON：后续自动粗剪与接口对接
- YAML：便于人工快速审阅和轻量配置化编辑，减少 JSON 冗余感
- PDF：归档与外部沟通

验收标准：

- 至少支持 2 种以上结构化导出格式，推荐同时支持 JSON 和 YAML。
- 不同导出内容的核心结构保持一致。
- JSON 与 YAML 字段可供二期自动剪辑模块直接消费。

## 5. 质量控制规则

## 5.1 强约束

- 所有分镜必须引用素材 ID。
- 未进入素材表的内容不得出现在分镜中。
- 地点归属不可混淆。
- 素材时长分配必须遵循信息量规则。
- 若使用具体 BGM，分镜时长分配必须校验节拍卡点合理性。

## 5.2 风险点

- 模型臆造素材
- 跨地点串镜
- 节奏平均化导致无爆点
- 分镜秒数与音乐节拍冲突
- 发布文案与视频主题脱节

## 5.3 质检清单

- 是否严格引用真实素材
- 是否满足目标时长
- 是否存在开头钩子
- 是否在 1/4、1/2、3/4 和结尾设置节奏节点
- 是否输出完整的节奏建议和发布方案
- 是否在已选 BGM 场景下完成节拍卡点校验

## 6. 一期上线验收标准

- 用户可创建项目并录入素材。
- 系统可生成 3 至 5 个主题方向。
- 系统可生成可执行分镜，且不创造不存在素材。
- 可导出 Markdown、JSON、YAML、PDF。
- 用至少 3 至 5 个真实项目完成回放验证。

## 7. 二期衔接要求

为支持后续自动粗剪，一期必须提前准备：

- 标准分镜 JSON Schema
- 对应的 YAML 模板示例
- 稳定的素材 ID 体系
- 明确的镜头时长、转场、字幕、BGM、节拍点字段
- 至少 2 套以上可复用的 Prompt 模板
- 至少 2 类以上规则库样例，例如平台规则库、节奏规则库或地点连续性规则库
