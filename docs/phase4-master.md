# Phase 4 主控文档：自动粗剪质量与平台化节奏增强

> 状态：已于 2026-07-17 完成当前范围。实现与验收结果见 `phase4-checklist.md`，轻量部署结论见 `phase4-deployment-storage.md`。

## 1. 文档目标

本文档用于定义四期开发目标、优先级、边界和实施顺序。四期不再以“补齐工作流”为目标，而是把当前可运行的导演助手升级为更稳定、更像剪辑师思考的自动初剪系统。

四期核心目标：

- 提升自动粗剪的可用度，让生成结果从“能看”接近“可继续精剪”。
- 将短视频节奏方法论产品化，支持不同平台和视频类型的节奏差异。
- 优化音频节拍识别与剪映踩点结果的偏差，为后续更准确卡点打基础。
- 优化 LLM 编排速度、稳定性和可恢复体验。
- 技术债跟随业务开发逐步偿还，不单独启动大规模重构。

## 2. 当前判断

### 2.1 暂缓复杂协作

当前主要使用者是单人自媒体创作者，后续朋友使用也以轻量共享为主。四期不做复杂协作系统，不做评论、审批、多人任务分派和细粒度权限矩阵。

保留现有用户、角色、登录和项目归属能力，作为后续扩展基础。

### 2.2 资产管理跟随部署方案

当前媒体文件仍以本地路径为主。是否同步云盘或对象存储，需要和上线部署、数据存储、成本预算一起设计。

四期暂不做重型 DAM，也不做完整媒体云同步。仅在上线规划中评估：

- 10 用户同时在线的轻量部署方案。
- SQLite 是否继续使用，或迁移到 Postgres。
- 本地素材路径、预览文件、关键帧、上传音频和导出草稿的存储边界。
- 对象存储接入时机。

### 2.3 自动粗剪是第一优先级

当前自动粗剪已经能生成结构，但存在以下问题：

- 画面吸引力不稳定。
- 音乐节奏与画面卡点不够准确。
- 生成分镜容易平均化，缺少短视频的持续刺激点。
- 转场、字幕、照片动效、口播和节奏推进仍然偏弱。

四期优先解决这些问题。

### 2.4 LLM 编排要更快、更稳、更顺

现有 LLM 能力已经覆盖主题、节奏、分镜和导出建议，但存在：

- 推理慢。
- Codex / OAuth 类模型受网络和登录状态影响，偶发掉线。
- 长任务等待感强。
- 失败后恢复和重试体验还不够顺滑。

四期要把 LLM 从“能调用”升级为“可持续使用”。

## 3. 剪辑节奏方法论

四期节奏引擎采用“平台 x 视频类型 x 目标时长”的三维模型，不再用同一套短视频快切规则覆盖所有平台。

### 3.1 时钟法则

时钟法则用于保证视频结构不散：

- 开头：给钩子或结果预告。
- 25%：第一次推进，确认主题或路线。
- 50%：第二次推进或反转，切换地点、情绪、视角或信息密度。
- 75%：高潮段，放置最强视觉、最强情绪或最有用信息。
- 结尾：回收主题，形成记忆点或行动建议。

### 3.2 射门集锦法则

短视频不是传统电影的“正反合”节奏，更接近“射门集锦”。用户需要持续获得新的注意力回报。

在短视频场景中：

- 60 秒内建议至少 4 个注意力节点。
- 每 12-15 秒应出现一次明显推进、反转、强视觉、强信息或强字幕。
- 注意力节点不是普通卡点，而是用户愿意继续看的理由。
- 关键镜头应同时尽量匹配内容注意力点和音乐节拍点。

### 3.3 平台差异

不同平台对节奏的期待不同：

| 平台/场景 | 推荐节奏模式 | 主要特点 |
| --- | --- | --- |
| 抖音 / 快手 | `highlight_reel` | 强钩子、快切、密集反转、强字幕 |
| 小红书短视频 | `seed_and_guide` | 视觉种草 + 信息密度，机位、避坑和体验感重要 |
| B 站攻略长视频 | `chapter_explainer` | 章节化、解释感、路线逻辑和信息完整性更重要 |
| B 站旅行 Vlog | `emotional_vlog` | 沉浸、人物陪伴感、氛围段落和情绪递进 |
| 视频号 / 朋友圈 | `stable_story` | 情绪稳定、叙事完整、标题清晰，不宜过碎 |
| YouTube 旅行内容 | `chapter_story` | 开头承诺、章节推进、长段落信息和故事线并重 |

### 3.4 节奏字段升级

四期建议新增或抽象以下概念：

- `rhythmProfile`：平台节奏画像，决定切镜密度、字幕密度、注意力间隔和结构模板。
- `attentionBeats`：内容注意力点，用于安排钩子、推进、反转、高潮和收尾。
- `musicBeatGrid`：音乐节拍网格，用于真正的卡点切镜。
- `cutDensity`：切镜密度，区分快切、中速、慢节奏。
- `subtitleDensity`：字幕密度，区分金句型、信息型、弱字幕。
- `motionPolicy`：照片动效和视频速度变化策略。

## 4. 分阶段开发计划

四期按 A/B 两段推进，避免一开始同时展开节奏、口播、LLM 后台任务和部署评估，导致改动面过大。

### 4.1 P4-A：自动粗剪质量主线

目标：优先解决“自动初剪不好看、不够踩点、不够像平台内容”的核心问题。

P4-A 只做会直接影响分镜和剪映草稿质量的基础能力：

- 平台节奏画像。
- 内容注意力点。
- 剪映节拍校准。
- 节点驱动分镜。
- 基础转场、照片动效、字幕和 BGM 处理。

P4-A 暂不做完整 TTS 口播、复杂 LLM 后台任务、一键全流程和部署存储迁移。

### 4.2 P4-B：生成体验与自动化增强

目标：在 P4-A 的节奏和分镜基础稳定后，继续提升自动化体验。

P4-B 包含：

- 自动口播与旁白音轨。
- LLM 后台任务、缓存、重试和恢复。
- 一键初剪方案。

### 4.3 P4-C：上线与轻量多人使用评估

目标：为后续朋友试用和小规模部署做准备。

P4-C 只做评估和必要基础设施准备，不在 P4-A 阶段提前引入重型资产管理或复杂协作。

## 5. P4-A 开发范围

### Sprint 19A：平台节奏画像与 attention beats

目标：让节奏规划先理解平台和视频类型，再生成适合的节奏结构。

交付：

- 新增平台节奏画像生成逻辑。
- 根据 `platform + videoType + targetDurationSec + stylePreference` 生成 `rhythmProfile`。
- 根据 `rhythmProfile` 生成 `attentionBeats`。
- 节奏页展示内容注意力节点和音乐节拍点的区别。
- 分镜生成读取 `attentionBeats`，优先把强素材放到注意力节点。

验收：

- 抖音 60 秒内容能生成 4 个左右注意力节点。
- B 站 5 分钟攻略能生成章节节点，而不是 15 秒一反转的快切结构。
- 分镜结果能说明每个关键段为什么放在该位置。

### Sprint 20A：剪映节拍对齐优化

目标：缩小系统节拍识别与剪映踩点结果的偏差。

当前问题：

- `librosa` 的 onset / beat_track 与剪映的“踩节拍 1 / 踩节拍 2 / 强弱拍”并不完全等价。
- 系统目前更像通用音频分析，缺少对剪映输出习惯的校准。
- 用户上传同一音频后，系统生成的切点和剪映内置识别存在明显错位。

交付：

- 增加“剪映对齐校准”能力。
- 支持用户手动输入或导入少量剪映参考节拍点，生成 offset / scale / density 校准参数。
- 在节奏页展示：原始识别点、剪映校准点、当前模式输出点。
- 保存音频分析元数据：`analysisEngine`、`detectedBpm`、`beatOffsetSec`、`calibrationSource`、`confidence`。
- 增加节拍偏移微调控件，支持整体前后移动节拍点。
- 增加节拍密度策略：粗、细、强拍、手动校准。

验收：

- 同一音频重新上传后能复用校准参数。
- 用户能通过少量手动校准让系统切点更接近剪映。
- 分镜生成使用校准后的节拍点，而不是原始识别点。

### Sprint 21A：节点驱动分镜编排

目标：从“素材排序生成分镜”升级为“内容节点驱动分镜”。

详细技术方案见 [`narrative-storyboard-planning.md`](narrative-storyboard-planning.md)。后续实现应以“路线主轴 + 叙事骨架 + 素材评分 + 约束搜索”为主，不再继续强化最近 `attentionBeat` 贴标签的粗粒度方案。

交付：

- 分镜增加 `attentionRole`：钩子、铺垫、推进、反转、高潮、缓冲、收尾。
- 素材先通过规则推断 `visualStrength`：强、中、弱。规则来源包括 `shotType`、`informationDensity`、`functionTags`、`emotionTags`、`visualTags`，后续再接入视觉模型自动评分。
- 分镜生成时优先把强视觉素材放入钩子、反转和高潮位。
- B 站攻略类内容优先按章节组织，信息素材优先级高于纯氛围素材。
- 抖音 / 小红书短视频优先保证前 3 秒钩子和 12-15 秒注意力点。

验收：

- 分镜列表能看出每个镜头的叙事功能。
- 同一批素材在抖音和 B 站平台下会生成不同节奏结构。
- 自动结果减少“平均铺素材”的问题。

### Sprint 22A：粗剪效果增强

目标：让导出的剪映草稿更接近可用初剪。

交付：

- 增加基础转场策略：硬切、淡入淡出、闪白、速度变化建议。
- 增加照片动效策略：慢推、轻缩放、横移、定格。
- 增加字幕策略：钩子字幕、信息字幕、反转字幕、收尾字幕。
- 增加 BGM fade in / fade out 和关键节点音量处理。
- 预留口播字段：`voiceoverText`、`voiceoverRole`、`voiceoverTiming`，四期可先不做 TTS。

验收：

- 导出的剪映草稿包含更明确的视频轨、字幕轨、音频轨和基础动效字段。
- 不追求自动成片，但要让剪辑师减少第一轮结构性返工。

## 6. P4-B 开发范围

### Sprint 22B：自动口播与旁白音轨

目标：支持把分镜字幕、导出文案或预设口播稿生成旁白音频，并写入后续粗剪导出链路。

当前规划状态：

- 四期原规划只预留了 `voiceoverText`、`voiceoverRole`、`voiceoverTiming` 字段。
- 尚未规划完整的 TTS Provider、口播稿生成、音频文件管理、音轨对齐和剪映导出。
- 本 Sprint 用于补齐自动口播能力，但不做未经授权的声音克隆。

前沿调研结论：

- 主流方案已经从“机械 TTS”升级为“可控情绪、语速、语气、停顿的神经语音合成”。
- OpenAI、Azure、Google、ElevenLabs 等都支持文本转语音；部分服务支持流式生成、SSML、音色选择、时间戳或自定义声音。
- 对本项目来说，第一阶段不需要实时语音对话，应优先做“离线生成口播音轨 + 时间线对齐 + 剪映草稿写入”。

交付：

- 新增 `voiceoverProvider` 配置，复用现有 LLM Provider 的“厂商无关”设计思路。
- 支持至少一种 API Key 类型的 TTS Provider。
- 支持三种口播来源：
  - 根据分镜字幕生成逐镜头口播。
  - 根据导出文案生成整段旁白。
  - 用户手动输入固定口播稿。
- 增加口播脚本结构：`voiceoverText`、`voiceoverStyle`、`voiceoverSpeed`、`voiceoverEmotion`、`voiceoverTiming`。
- 生成音频后保存 `voiceoverAudioPath`、`durationSec`、`providerMeta`。
- 在分镜或导出页提供试听、重新生成、删除口播音频。
- 剪映草稿导出支持新增 voiceover 音轨，并与 BGM 做音量避让。
- 字幕与口播默认不强制一致，允许“字幕短、口播长”的内容形式。

验收：

- 用户能把当前分镜字幕生成一条口播音频。
- 用户能把预设文案生成一条完整旁白音频。
- 导出的剪映草稿包含 voiceover 音轨。
- BGM 与 voiceover 同时存在时，BGM 音量自动降低，避免口播被盖住。
- 没有 TTS Provider 配置时，页面明确提示需要配置，不影响原有导出。

边界：

- 四期不做未授权声音克隆。
- 四期不承诺真人级配音质量。
- 四期不做实时语音对话。
- 自定义音色、声音克隆、跨语言配音和角色多音色，进入后续 backlog。

### Sprint 23B：LLM 性能与稳定性

目标：减少等待、掉线和重复推理成本。P4-A 阶段只保留轻量 fallback 和错误提示，完整后台任务放到 P4-B。

交付：

- LLM 任务后台化，支持任务状态轮询。
- 长任务支持取消、重试和恢复。
- Prompt 拆段：结构先行，文案后补。
- 对主题、节奏、分镜、导出建议做输入 fingerprint 缓存。
- LLM 失败时保留规则 fallback，并清楚说明使用了 fallback。
- Codex / OAuth 掉线时给出重新连接和恢复任务入口。

验收：

- 用户离开页面再回来能看到任务状态。
- 同样输入重复生成时可以复用缓存或提示无需重复推理。
- LLM 失败不会阻断主工作流。

### Sprint 24B：一键初剪方案

目标：降低操作步骤，让用户可以快速得到一版可检查的初剪方案。

交付：

- 新增“一键生成初剪方案”入口。
- 串联主题、节奏、分镜、导出建议。
- 每步产物可单独重跑。
- 支持只重生成当前段或当前节点。
- 对比新旧版本，用户确认后应用。

验收：

- 新项目录入素材后，可以一键生成完整初剪方案。
- 用户仍可逐页进入每个环节微调。

## 7. P4-C 开发范围

### 部署与数据存储评估

目标：为 10 用户同时在线做轻量上线准备。

评估项：

- SQLite 是否继续使用，或迁移 Postgres。
- 音频、预览文件、关键帧、剪映草稿是否需要对象存储。
- 本地素材路径在多人环境下的限制。
- 后台任务是否需要队列。
- LLM 调用限流、费用统计和失败重试。

建议：

- 四期前半段继续单机 SQLite。
- 如果开始多人真实使用，再进入 Postgres + 对象存储方案。
- 暂不做重型资产管理平台。

### 轻量协作

仅保留：

- 用户管理。
- 项目归属。
- 项目可见性。

暂不做：

- 评论。
- 审批。
- 多人同时编辑。
- 复杂权限矩阵。

## 8. 数据结构预告

P4-A 优先落地最小字段集：

- `rhythm_profile_json`
- `attention_beats_json`
- `beat_calibration_json`
- `audio_fingerprint`
- `audio_analysis_version`
- `storyboard_segment.attention_role`
- `storyboard_segment.visual_strength`
- `storyboard_segment.motion_policy`
- `storyboard_segment.transition_policy`

P4-B 再落地口播和 LLM 后台任务字段：

- `storyboard_segment.voiceover_text`
- `storyboard_segment.voiceover_style`
- `storyboard_segment.voiceover_audio_path`
- `voiceover_provider_config`
- `voiceover_generation_job`
- `llm_task`
- `llm_result_cache`

实际 migration 以开发时 schema 设计为准，每次字段变更同步 `schema-migration.md` 和 `api.md`。

## 9. 四期边界

四期不做：

- 自动渲染完整成片。
- 重型云端 DAM。
- 平台盗版 BGM 下载。
- 网页 Cookie 抓取式登录。
- 复杂多人协作。
- 大规模分布式任务系统。

四期可以做：

- 更准确的节奏点。
- 更合理的平台化分镜。
- 更好的剪映草稿结构。
- 更稳定的 LLM 编排体验。
- 为部署上线做轻量预研。

## 10. 阶段验收标准

### 10.1 P4-A 验收

P4-A 完成时，应满足：

- 同一项目选择不同平台时，节奏规划和分镜结构明显不同。
- 抖音 / 小红书短视频能体现钩子、推进、反转、高潮和收尾。
- B 站攻略类内容能体现章节化和信息完整性。
- 上传音频后可通过校准让节拍点更接近剪映。
- 分镜生成能同时参考内容注意力点和音乐节拍点。
- 剪映草稿包含更明确的转场、动效、字幕和音频处理建议。

### 10.2 P4-B 验收

P4-B 完成时，应满足：

- 可选生成口播音轨，并能写入剪映草稿。
- LLM 任务失败、掉线、重试和恢复体验更稳定。
- 新项目录入素材后，可以一键生成一版完整初剪方案。

### 10.3 P4-C 验收

P4-C 完成时，应满足：

- 明确 10 用户同时在线的轻量部署方案。
- 明确 SQLite、Postgres、对象存储和后台任务队列的取舍。

## 11. 收尾结论

- P4-A 已完成平台化节奏、注意力节点、路线叙事编排、剪映节拍校准及草稿效果增强。
