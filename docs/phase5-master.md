# Phase 5 主控文档：发布质量闭环与成片智能优化

> 状态：规划完成，待开发。P4 已于 2026-07-17 关闭，P5 不再扩充基础工作流，而是提升真实成片质量、自动生成稳定性和持续优化能力。

## 1. 阶段定位

P4 已完成“素材录入 -> 平台化节奏 -> 路线叙事分镜 -> 口播 -> 剪映草稿 -> 本地生产部署”的完整闭环。当前系统已经能稳定产出可播放的粗剪，但距离高概率可发布仍有四类差距：

- 镜头是否抓眼仍主要依赖人工标签和通用 Vision 描述。
- 音乐节拍可校准，但自动结果与剪映、人工听感仍存在差距。
- 分镜同时受路线、叙事、节拍、口播、素材时长等约束，局部最优不等于整片最优。
- 生成了多个版本以后，系统还不能客观解释哪一版更好，也没有利用用户修改结果持续校准。

P5 的核心目标是建立质量闭环：

```text
素材分析 -> 多目标编排 -> 自动质量评分 -> A/B 对比 -> 人工修正 -> 参数校准
```

P5 的北极星不是增加按钮数量，而是让“第一版粗剪可继续使用”的概率持续提升。

## 2. 已锁定产品决策

### 2.1 面向单人创作者优化

当前主要用户仍是单人自媒体创作者，少量朋友试用不构成复杂协作需求。P5 不做评论、审批、多人实时编辑和复杂权限矩阵。

### 2.2 保持本地媒体与轻量部署

原始媒体继续保存在本地，SQLite WAL、本地文件和 Windows 生产部署继续作为默认方案。只有满足以下任一条件时，才启动 Postgres、对象存储和外置任务队列迁移：

- 同时在线用户稳定超过 10 人。
- 需要跨设备访问同一批原始素材。
- 单机任务排队明显影响交互。
- 本地存储和备份无法满足可靠性要求。

### 2.3 不做端到端黑盒剪辑

LLM 可以建议主题、叙事骨架、候选镜头、字幕和口播，但不能绕过以下硬约束：

- 素材 ID 和文件必须真实存在。
- 路线型内容正文默认不能无解释地倒退。
- 时间线不能重叠或超出可用素材时长。
- 口播必须完整，字幕必须与最终口播一致。
- 生成结果必须保留选择理由和配置快照。

### 2.4 剪映仍是最终精剪工作台

P5 继续输出剪映草稿，不自建完整 NLE，也不承诺自动渲染最终成片。系统负责减少结构性劳动，用户负责最终审美判断和发布前精修。

## 3. P5 目标与指标

### 3.1 质量指标

P5 首先建立基线，再按真实数据调整目标。建议初始验收指标如下：

| 指标 | 定义 | P5 目标 |
| --- | --- | --- |
| 粗剪可用率 | 生成后无需推倒重来，可直接进入局部精修的版本比例 | 基准项目达到 70% |
| 路线连续性 | 正文路线无无理由倒退的项目比例 | 100% |
| 注意力覆盖 | 关键注意力槽位有合适素材覆盖的比例 | 不低于 90% |
| 口播完整率 | 无截断、无字幕与发音文本不一致 | 100% |
| 节拍参考误差 | 与人工/剪映参考点匹配后的中位时间误差 | 基线下降 30% 以上 |
| 剪映草稿打开率 | 草稿可正常发现、打开且主要轨道存在 | 不低于 95% |
| 生成可追溯率 | 自动镜头具备来源、评分和选择理由 | 100% |

### 3.2 性能与稳定性指标

- 规则生成和质量评分不依赖 LLM，可离线完成。
- LLM 失败时保留已完成产物，并允许按原参数恢复。
- 相同输入和配置可复用分析缓存。
- 每次完整生成保存 Provider、模型、Prompt 版本、算法版本和参数快照。
- 生产部署升级失败时能自动回滚到上一健康版本。

## 4. 总体技术架构

P5 在现有服务上增加五个可独立测试的领域模块：

```text
AssetFeatureExtractor
        |
        v
NarrativePlanner -> TimelineOptimizer -> QualityEvaluator
        ^                  |                    |
        |                  v                    v
   LLM semantic hint   GenerationVersion   CreatorFeedback
```

### 4.1 AssetFeatureExtractor

负责提取不依赖具体主题的素材特征，并按文件 fingerprint 缓存：

- 技术质量：清晰度、曝光、抖动、黑帧、静帧、可用时长。
- 视觉强度：主体显著性、运动强度、色彩对比、人物/景观占比。
- 剪辑属性：景别、方向、运动趋势、推荐入点/出点、转场兼容性。
- 语义属性：地点、主体、动作、情绪、信息价值。
- 去重属性：感知哈希、相邻镜头相似度、重复构图。

本地可计算项优先使用 ffmpeg/OpenCV 类确定性能力；语义描述再使用当前生效的 Vision Provider。Vision 失败不阻断本地分析和手工录入。

### 4.2 NarrativePlanner

根据 `platform + videoType + targetDuration + route + theme` 生成叙事槽位，不直接决定最终素材：

- 抖音/快手：钩子、推进、反转、展开、高潮、回收。
- 小红书：视觉种草、体验证明、信息价值、收藏理由。
- B 站攻略：问题承诺、章节、证据、细节、总结。
- 旅行 Vlog：进入、沉浸、关系/情绪推进、高潮、余味。

LLM 可以生成槽位说明和语义需求；规则层负责槽位顺序、数量、时间预算和平台下限。

### 4.3 TimelineOptimizer

沿用当前 Beam Search，但从单一累计分数升级为“硬约束 + 多目标评分 + 修复器”。

硬约束：

- 路线单调性及允许的开头预告例外。
- 素材存在、默认不重复、时长合法。
- 总时长与口播可读时长边界。
- 关键槽位不能被补时桥段挤出。
- 相邻镜头不能使用高相似度重复构图。

软目标：

```text
score = narrative_fit
      + semantic_relevance
      + visual_strength
      + route_continuity
      + beat_alignment
      + shot_diversity
      + transition_compatibility
      + information_value
      - repetition_penalty
      - voiceover_overflow_penalty
```

不同平台使用不同权重，而不是共享一套固定排序。优化器保留 Top-K 候选时间线，供质量评估和 A/B 对比使用。

### 4.4 QualityEvaluator

质量评分必须独立于生成器，避免生成器给自己打满分。首期采用可解释的确定性评分：

- `routeScore`：路线连续性与地点覆盖。
- `narrativeScore`：槽位完整性、反转间隔、高潮位置、结尾回收。
- `rhythmScore`：镜头边界与节拍/段落节点误差。
- `visualScore`：强镜头覆盖、重复构图和景别多样性。
- `voiceoverScore`：发音完整、字幕一致和画面对齐。
- `exportScore`：文件存在、轨道完整、草稿结构合法。

总分只用于排序和诊断，不代替用户判断。页面必须同时显示分项分数和扣分原因。

### 4.5 CreatorFeedback

P5 不训练自有大模型，先采集低成本、高价值反馈：

- 接受整版。
- 删除镜头。
- 替换镜头。
- 调整顺序。
- 调整时长。
- 修改字幕/口播。
- 标记“路线错误、节奏拖沓、画面不够强、口播过密”等原因。

反馈先用于调整项目级和平台级权重，不直接在线修改全局算法，避免单个项目污染所有内容。

## 5. 音乐结构与踩点升级

P5 不再把所有 onset 都视为同一种节拍点。音频分析输出四层结构：

- `onsetPoints`：细粒度起音点，适合快切候选。
- `beatGrid`：稳定拍点网格。
- `downbeatPoints`：小节强拍，适合关键切换。
- `sectionBoundaries`：前奏、主歌、副歌、间奏、尾奏等结构变化候选。

自动分镜优先级：

1. 钩子、反转、高潮优先吸附强拍或段落边界。
2. 普通镜头吸附邻近 beat，不为卡点牺牲最低可读时长。
3. 口播镜头优先保证完整发音，再在容差内寻找节拍。
4. 照片和低运动素材允许跨拍停留，避免机械一拍一切。

节拍引擎继续使用可插拔接口：现有 librosa/能量算法为 baseline，P5-A 先建立参考数据集，再决定是否引入额外深度节拍模型，避免先增加重依赖却无法证明效果提升。

## 6. 分阶段实施计划

### Sprint 25：质量基线与黄金项目集

目标：先解决“好不好没有统一标准”的问题。

交付：

- 建立 5-10 个黄金项目，至少覆盖抖音旅行短片、小红书种草、B 站攻略和旅行 Vlog。
- 天台山作为固定回归项目，冻结素材元数据、BGM 参考点和人工认可时间线。
- 新增离线 benchmark 命令，批量生成并输出质量报告。
- 保存算法版本、模型版本、参数快照和分项评分。
- 前端版本详情展示质量摘要与扣分原因。

### Sprint 26：素材质量、镜头边界与去重

目标：提高“选什么镜头”的质量。

交付：

- 批量素材分析任务与 fingerprint 缓存。
- 技术质量、运动强度、视觉显著性和相似度特征。
- 自动推荐素材可用区间，而不是默认从文件开头取片段。
- 重复素材、相似构图和低质量素材告警。
- 素材列表展示质量摘要，允许用户覆盖自动判断。

### Sprint 27：多约束时间线优化器

目标：把路线、叙事、视觉、节拍、口播和时长纳入同一次全局求解。

交付：

- 多目标权重配置与平台预设。
- Beam Search 状态向量升级。
- 相似镜头惩罚、关键素材保留、口播时长约束。
- Top-K 时间线候选和选择追溯。
- 约束失败时输出可操作建议，而不是静默生成异常结果。

### Sprint 28：音乐段落与视听对齐

目标：让关键内容节点不仅踩拍，还踩在正确的音乐结构上。

交付：

- 四层音乐结构数据模型。
- 节拍参考标注和误差评估工具。
- 自动 offset、tempo drift 和密度校准。
- 强拍/段落边界驱动钩子、反转和高潮。
- 节奏页可视化原始点、校准点、强拍和段落边界。

### Sprint 29：口播、字幕与画面联合排程

目标：避免分镜先生成、口播后硬塞造成的倍速过快和尾音截断。

交付：

- 叙事槽位阶段先生成口播预算，再分配镜头时长。
- 真实 TTS 发音边界反向修正镜头停留时间。
- 超出平台舒适语速时自动精简文案或建议延长画面。
- BGM ducking 区间跟随真实口播边界。
- 最终字幕继续与口播共用同一文本和时间边界。

### Sprint 30：A/B 版本实验与创作者反馈

目标：让不同模型、算法参数和平台预设可以低成本比较。

交付：

- 同一项目一键生成 A/B 两个版本。
- 对比镜头顺序、素材使用、节拍、口播、质量分和生成成本。
- 支持逐段选择 A 或 B，并合并为新版本。
- 记录接受、替换、删除、调序等反馈。
- 项目级偏好自动形成可恢复的权重预设。

### Sprint 31：剪映导出验证与生产稳定性

目标：把“生成成功”升级为“剪映中可验证地打开并继续编辑”。

交付：

- 导出前检查媒体路径、口播、BGM、字幕和草稿结构。
- 导出后生成 manifest 和诊断报告。
- 增加素材重新定位能力，处理媒体根目录移动。
- 加固 Windows 隐藏任务、健康检查、升级回滚和 OAuth 回调诊断。
- 将真实 Provider smoke 与纯 Mock CI 分层运行。

### Sprint 32：P5 验收与冻结

目标：用黄金项目证明质量提升，而不是只证明接口可用。

交付：

- 完整 benchmark 对比 P4 基线与 P5 结果。
- 天台山及其他平台项目人工验收。
- 性能、成本、失败恢复和生产升级报告。
- 更新 PRD、SOP、API、迁移文档和部署文档。

## 9. 测试策略

### 单元测试

- 特征提取边界、相似度和可用区间。
- 每项硬约束及平台权重。
- 质量评分和扣分原因。
- 节拍误差、段落吸附和口播时长预算。

### 属性测试

对随机素材集合验证：

- 时间线不重叠。
- 素材不存在时不会进入结果。
- 默认不重复素材。
- 路线正文不倒退。
- 关键槽位不会被补时挤出。
- 口播不会被时间线截断。

### 黄金项目回归

- 固定输入、算法版本和随机种子。
- 允许文案变化，但结构指标不得退化。
- 每次优化输出 P4 基线、当前 main 和候选分支三方报告。

### 生产 smoke

- 登录、Provider 连通、完整生成、Edge TTS、剪映草稿写入。
- 不在普通 CI 使用真实付费 Provider；生产 smoke 独立手动或定时执行。

## 10. 风险与控制

| 风险 | 控制方案 |
| --- | --- |
| 质量分被算法“刷分” | 评分器独立、黄金项目人工复核、保留分项原因 |
| Vision 成本和延迟增长 | fingerprint 缓存、本地特征优先、批量任务可取消 |
| 多目标参数越来越复杂 | 平台预设为主，高级参数默认折叠，支持恢复默认 |
| 节拍深度模型依赖过重 | 先 benchmark，再以可插拔 optional extra 接入 |
| 用户反馈过少无法学习 | 先做项目级偏好，不承诺模型训练和全局自学习 |
| 剪映格式变化 | adapter + manifest + 验证器，保留 5.9 明文兼容路径 |
| LLM/OAuth 网络不稳定 | 规则 fallback、任务恢复、API Key 备用、状态诊断 |

## 11. P5 不做

- 自建完整视频编辑器。
- 自动发布到抖音、B 站或小红书。
- 未授权声音克隆和数字人。
- 重型云端素材管理和多人审片。
- 以播放量作为实时强化学习奖励。
- 在没有黄金项目验证前引入大型分布式基础设施。

## 12. 实施顺序与预估

推荐严格按 `25 -> 26 -> 27 -> 28 -> 29 -> 30 -> 31 -> 32` 推进。质量基线必须先于算法升级，否则无法判断“感觉更智能”是否真的改善成片。

按单人开发并使用 AI 辅助估算：

- P5-A（Sprint 25-28）：4-6 周。
- P5-B（Sprint 29-30）：2-3 周。
- P5-C（Sprint 31-32）：1-2 周。
- 总计：约 7-11 周，真实 Provider 和剪映人工验证时间另计。

若需要压缩范围，最小可交付版本只做 Sprint 25、27、28、32；素材深度分析、A/B 合并和反馈预设可以后移，但不能跳过质量基线。
