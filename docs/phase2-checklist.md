# 二期开发前 Checklist

## 1. 文档目的

本文档用于在二期功能开发前，对当前一期实现进行技术债梳理与改造优先级归纳，帮助后续开发按“先稳底座，再扩能力”的方式推进。

适用对象：

- 产品
- 开发
- 测试
- 后续协作剪辑流程设计人员

---

## 2. 当前判断

当前一期主链路已经可用，能够完成以下工作：

- 项目创建与编辑
- 素材录入与维护
- 主题生成与选择
- 节奏规划
- 分镜生成与编辑
- 导出预览与下载

但从二期扩展角度看，仍存在一批需要提前收口的技术债。如果不先整理，二期接入音频分析、真实节拍识别、LLM 建议生成和多用户能力时，改造成本会明显上升。

---

## 3. P0：二期开工前建议先完成

这些事项建议在正式启动二期功能开发前优先处理。

### 3.1 统一文本编码

现状：

- 前后端部分文件仍残留历史乱码文案。
- 文档、页面文案、导出内容的中文可维护性较差。

风险：

- 后续修改 PRD 对应逻辑时难以精确定位。
- 页面、导出、接口说明可能出现显示不一致。

建议动作：

- 统一仓库文本文件编码为 `UTF-8`。
- 清理前后端关键业务页面与文档中的乱码字符串。
- 对导出模板的中文内容做一次专项检查。

验收标准：

- `docs`、前端页面文案、后端返回消息均可正常显示中文。

### 3.2 补显式素材排序字段

现状：

- 当前素材顺序主要依赖自动生成的素材 ID 末尾序号推导。
- 这能满足一期使用，但属于隐式排序规则。

风险：

- 二期若引入拖拽排序、批量导入、地点重排，排序逻辑容易失控。
- “同功能标签按录入顺序排列”的业务规则会变得脆弱。

建议动作：

- 在素材表增加 `sort_order` 字段。
- 查询素材列表时统一按 `sort_order` 排序。
- 新增素材默认追加到末尾。

验收标准：

- 不依赖素材 ID 也能稳定恢复用户录入顺序。

### 3.3 拆分数据读写与业务生成逻辑

现状：

- `repository.py` 同时承担数据读写、规则计算、生成逻辑拼装。

风险：

- 二期功能继续叠加后，核心文件会快速膨胀。
- 后续排查问题时难以判断是数据层问题还是业务层问题。

建议动作：

- Repository：只负责数据库读写。
- Service：负责业务编排。
- Generator：负责主题、节奏、分镜生成规则。

验收标准：

- 二期新增生成逻辑时，不需要继续把大量规则塞进 Repository。

### 3.4 确定数据库迁移机制

现状：

- 当前字段仍在演进中。
- 二期大概率还会新增用户、音频分析、排序、AI 建议相关字段。

风险：

- SQLite 数据结构迭代会越来越难维护。
- 历史数据容易与新逻辑不兼容。

建议动作：

- 明确数据库 schema 迁移方案。
- 形成“新增字段 / 改字段 / 初始化默认值”的统一流程。

验收标准：

- 后续数据库结构变更可重复执行、可追溯。

---

## 4. P1：建议在二期早期尽快完成

这些事项可以和二期主功能并行推进，但越早做越省成本。

### 4.1 登录与权限改造成正式方案

现状：

- 登录能力已经起步，但仍偏演示态。
- 后续用户体系已明确要转到后端数据库管理。

风险：

- 多用户场景下，项目与用户之间缺少稳固边界。
- 登录、鉴权、项目归属容易出现后续返工。

建议动作：

- 用户信息统一放到后端数据库。
- 使用密码哈希而不是明文。
- 增加登录态过期、登出、接口鉴权。
- 项目与用户建立 owner 关系。

验收标准：

- 项目只能由登录用户访问自己的数据。

### 4.2 节奏规划升级为可扩展结构

现状：

- 当前“根据项目生成节奏规划”是规则推导，不是真实音频分析。

风险：

- 二期接音频上传和自动 beat detection 时，现有字段可能不够表达来源与状态。

建议动作：

- 为节奏规划补充“数据来源”概念：
  - 手工填写
  - 规则生成
  - 音频分析生成
- 预留音频文件路径、分析状态、分析结果版本字段。

验收标准：

- 后续支持音频上传后，不需要推翻现有节奏页数据结构。

### 4.3 分镜排序机制升级为双层结构

现状：

- 一期按功能标签优先级 + 素材录入顺序生成分镜。
- 这符合一期目标，但不适合直接扩展到 AI 排序。

风险：

- 二期接入大模型后，AI 顺序和保底业务规则容易互相覆盖。

建议动作：

- 保留基础规则排序作为 fallback。
- 新增 AI 建议排序层。
- 人工调整结果单独持久化，不覆盖原始建议。

验收标准：

- 规则排序、AI 建议、人工修订三层职责清晰。

### 4.4 自动化测试补齐

现状：

- 当前主要依赖人工回归。

风险：

- 二期功能一多后，每次改动都会影响已有 workflow。

建议动作：

- 补后端规则单测：
  - 分镜排序
  - 分镜校验
  - 节奏点依赖
- 补前端关键流程测试：
  - 项目 CRUD
  - 素材 CRUD
  - 分镜编辑跳转
- 补一条端到端冒烟流程：
  - 新建项目 → 录入素材 → 主题 → 节奏 → 分镜 → 导出

验收标准：

- 核心 workflow 有最基本的自动回归保障。

---

## 5. P2：可在二期迭代中逐步完善

这些事项不是二期开工阻塞项，但建议纳入二期范围内逐步完成。

### 5.1 分镜校验继续细化

建议补充：

- 镜头时长上下限校验
- 路线跳回告警
- 空功能镜头比例提醒
- 节拍容差参数化
- 导出前完整质检结果

### 5.2 导出模块增强

建议补充：

- 导出模板版本号
- 文件命名规范
- Markdown / JSON / YAML 一致性检查
- 导出快照留档

### 5.3 页面交互状态统一

建议补充：

- 统一 loading 体验
- 统一保存成功提示
- 统一失败重试提示
- 未保存离开提醒

### 5.4 文档补“当前实现状态”章节

建议在 PRD / SOP / API 文档中增加：

- 已实现内容
- 当前仍为规则 mock 的部分
- 明确延期到二期的部分

这样后续协作时，不会把一期演示逻辑误认为已完成的正式能力。

---

## 6. 建议排期方式

建议按以下顺序推进：

1. 先完成 P0：编码、排序字段、分层、迁移方案
2. 再启动 P1：认证、节奏结构升级、分镜排序升级、测试
3. 最后在二期开发过程中逐步吸收 P2

---

## 7. 与二期能力的对应关系

| 二期目标 | 先决准备 |
| --- | --- |
| 音频上传与自动节拍识别 | 节奏规划数据结构升级、迁移机制 |
| LLM 参与主题/分镜建议 | 生成逻辑分层、排序机制双层化 |
| 多用户协作 | 用户数据库、鉴权、项目 owner |
| 更稳定的时间线生成 | 显式素材排序、分镜校验增强、自动化测试 |
| 更正式的导出交付 | 导出版本化、模板一致性、快照留档 |

---

## 8. 最终结论

当前项目已经具备进入二期的基础，但不建议直接在现有结构上无整理地堆功能。

更合理的方式是：

- 先补齐底层可扩展能力
- 再启动音频分析、真实节拍识别、LLM 建议生成

这样可以显著降低二期返工成本，并保持一期已完成 workflow 的稳定性。
